## 1. Project Overview
This project implements a **GRPO (Generalized Reward PPO)** reinforcement learning pipeline based on the **ROLL** distributed framework. The core objective is to train a **Vision-Language Model (VLM)** to perform complex image editing tasks. The workflow follows a **VLM → CoT → DiT** paradigm:

* **VLM (Vision-Language Model) as Actor:** The primary target model being trained. It receives a user instruction and an original image to generate a **Chain-of-Thought (CoT)** text. This CoT provides detailed reasoning on how the image should be edited to satisfy the request.
* **CoT → DiT (Diffusion Image Transformer):** The CoT generated by the VLM serves as a prompt for a pre-trained image editing model (**DiT**, specifically *Qwen-Image-Edit*), which produces the final edited image.
* **VLM as Judge:** A powerful, independent VLM (e.g., *Qwen3-VL-30B*) acts as the **Reward Model**. It evaluates the edited image across multiple dimensions (consistency, reasoning accuracy, visual quality) to calculate a reward signal.
* **GRPO Training:** Utilizing this reward signal, the GRPO algorithm optimizes the **Actor VLM’s** policy, enabling it to generate higher-quality CoT that leads to superior visual outcomes from the DiT model.

## 3. Core Training Workflow

Each training iteration (`global_step`) follows these five critical stages:

### Phase 1: Experience Generation (Rollout)
* **Resource Shifting:** The `actor_train` and `critic` weights are **offloaded** to the CPU to free up VRAM.
* **Inference:** The `actor_infer` (a dedicated inference copy of the Actor VLM) is loaded onto GPUs. It processes prompts (image + instruction) from the dataset to generate a batch of CoT responses.
* **Dispatch:** The generated CoT and associated metadata are sent to the `rewards` cluster.

### Phase 2: Reward Computation
The `ImageEditRewardWorker` executes the following:
1.  **CoT → DiT:** Loads the **Editor** model and uses the CoT as a prompt to generate the edited image.
2.  **Image Scoring:** Offloads the Editor and loads the **Judge VLM** to score the image across various dimensions.
3.  **Return Signal:** Sends the final weighted scores (`response_level_rewards`) back to the main pipeline.

### Phase 3: Advantage Estimation
The main pipeline collects the rewarded experience data and prepares for PPO/GRPO updates:
* **Reloading Models:** `actor_train` and `critic` are moved back to the GPUs.
* **Log Prob & Value Calculation:**
    * `reference` cluster calculates reference policy log-probs (`ref_log_probs`).
    * `actor_train` calculates current policy log-probs (`old_log_probs`).
    * `critic` calculates state values (`values`).
* **Advantage Calculation:** Computes advantage values and returns using **GAE (Generalized Advantage Estimation)**.

### Phase 4: Model Update
* The `actor_train` and `critic` clusters perform multiple **PPO epochs** to update model parameters based on the calculated advantages.
* New weights from `actor_train` are periodically synchronized with `actor_infer` to ensure the next rollout uses the latest policy.

### Phase 5: Logging & Checkpointing
System metrics are recorded, and model checkpoints are saved before the pipeline proceeds to the next `global_step`.
